# Elasticsearch 文档命令

# 更新文档

参考链接：https://www.elastic.co/guide/en/elasticsearch/reference/current/docs-update.html

# 使用 Script 修改文档

一个更新脚本具有以下三项重要元素：

  • script 的作用比较强大，可以修改字段值，往列表中添加新元素，执行更复杂的 painless 脚本等等。 默认的脚本语言是 painless。
  • 由于更新要获得现有文档的 _source 内容，修改并重新索引新的文档，因此脚本会修改 _source 中的字段。使用 ctx._source 来引用 _source，使用 ctx._source[字段名] 来引用某个指定的字段。
  • 如果需要变量，推荐在 params 下作为参数单独定义，和脚本本身分开。这是因为脚本需要编译，一旦编译完成，就会被缓存。如果使用不同的参数，多次运行同样的脚本，脚本只需要编译一次。之后的运行都会从缓存中获取现有的脚本。相比每次不同的脚本，这样运行会更快，因为不同的脚本每次都需要编译。这个思想和 Oracle 的绑定变量与软编译概念异曲同工。
#先插入一条数据，指定 id 为 1
PUT my-index/_doc/1
{
  "name": "chengzw",
  "age": 18
}

#修改这条文档的值，加上一个字段 new_field 
POST my-index/_update/1
{
  "script" : "ctx._source.new_field = 'new content'"
}

#查看文档修改后的值
GET my-index/_doc/1

#返回结果
{
  "_index" : "my-index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 2,
  "_seq_no" : 1,
  "_primary_term" : 1,
  "found" : true,
  "_source" : {
    "name" : "chengzw",
    "age" : 18,
    "new_field" : "new content"
  }
}

# 使用 doc 修改文档值

#先插入一条数据，指定 id 为 1
PUT my-index/_doc/1
{
  "name": "chengzw",
  "age": 18
}

#指定修改 name 的值
POST my-index/_update/1
{
  "doc":{
    "name": "jack"
  }
}
#相同作用的 script 写法
POST my-index/_update/1
{

  "script": "ctx._source.name = 'peter'"
}


#查看修改后的值
GET my-index/_doc/1

#返回结果
{
  "_index" : "my-index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 2,
  "_seq_no" : 1,
  "_primary_term" : 1,
  "found" : true,
  "_source" : {
    "name" : "jack",
    "age" : 18
  }
}

如果 doc 和 script 同时作用于同一个字段，那么 doc 将会被忽略。

# noop 返回结果

如果 update 最终没有修改原文档的任何值，返回结果 result 将会显示 noop。

#先插入一条数据，指定 id 为 1
PUT my-index/_doc/1
{
  "name": "chengzw",
  "age": 18
}

#修改 name 的值，值和原先一样
POST my-index/_update/1
{
  "doc": {
    "name": "chengzw"
  }
}

#返回结果
{
  "_index" : "my-index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 3,
  "result" : "noop",  #没有修改
  "_shards" : {
    "total" : 0,
    "successful" : 0,
    "failed" : 0
  },
  "_seq_no" : 2,
  "_primary_term" : 1
}

想要 update 没有修改文档的时候 result 也显示 updated，可以加上 "detect_noop": false。

POST my-index/_update/1
{
  "doc": {
    "name": "chengzw"
  },
   "detect_noop": false
}

#返回结果
{
  "_index" : "my-index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 6,
  "result" : "updated",
  "_shards" : {
    "total" : 2,
    "successful" : 1,
    "failed" : 0
  },
  "_seq_no" : 5,
  "_primary_term" : 1
}

# Upsert

为了处理更新时文档并不存在的情况，可以使用 upsert。这个单词是关系数据库中 update 和 insert 的混成词。如果被更新的文档不存在，可以在 json 的 upsert 部分中添加一个初始文档用于索引。

POST test/_update/1
{
  "doc": {
    "counter": 3
  },
  "upsert": {
    "counter": 3,
    "name": "chengzw"
  }
}

# Upsert 配合 script

配合 script 使用，如果文档不存在，则 upsert 元素的内容将作为新文档插入，如果文档存在，则执行 script 中的内容。

POST test/_update/1
{
  //count + 4
  "script": {
    "source": "ctx._source.counter += params.count",
    "lang": "painless",
    "params": {
      "count": 4
    }
  },
  //设置 count 为 1
  "upsert": {
    "counter": 1
  }
}

cripted upsert 不管文档是否存在，都执行 script。

POST test/_update/1
{
  "scripted_upsert": true,  //不管文档是否存在，都执行 script。
  "script": {
    "source": "ctx._source.counter += params.count",
    "lang": "painless",
    "params": {
      "count": 4
    }
  },
  "upsert": {
    "counter": 1
  }
}

#文档是不存在的，count 最终是 5
GET test/_doc/1

#返回结果
{
  "_index" : "test",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 3,
  "_seq_no" : 10,
  "_primary_term" : 1,
  "found" : true,
  "_source" : {
    "counter" : 5
  }
}

# Doc upsert

当文档存在时修改 name 的值，当文档不存在时插入新文档。

POST test/_update/1
{
  "doc": {
    "name": "new_name"
  },
  "doc_as_upsert": true
}

# 并发控制

# 使用 _version 字段进行并发控制（乐观锁）-- 6.7 版本以前

插入一条文档：

PUT /test_index/_doc/1
{
  "test_field":"test test"
}

其中一个客户端先更新了一下这个数据，更新成功，version 有 1 变为 2.

PUT /test_index/_doc/1?version=1
{
    "test_field": "test client 1"
}

另外一个客户端，尝试基于 version=1 的数据去修改，，此时就会报错 version conflict：

PUT /test_index/_doc/1?version=1
{
    "test_field": "test client 2"
}

# 使用 if_seq_no 和 if_primary_term

if_primary_term 记录的就是具体的哪个主分片，而 if_seq_no 这个参数起的作用和旧版本中的 _version 是一样的，之所以加上 if_primary_term 这个参数主要是提高并发的性能以及更自然，因为每个 document 都只会在某一个主分片中，所以由所在主分片分配序列号比由之前通过一个参数 _version，相当于由整个 Elasticsearch 集群分配版本号要来的更好。

插入一条数据：

PUT /test_index/_doc/1
{
  "test_field":"test test"
}
{
  "_index" : "test_index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 3,
  "result" : "created",
  "_shards" : {
    "total" : 2,
    "successful" : 1,
    "failed" : 0
  },
  "_seq_no" : 2,
  "_primary_term" : 1
}

其中一个客户端，先更新了一下这个数据，_seq_no 会变为 3：

PUT /test_index/_doc/1?if_seq_no=2&if_primary_term=1
{
  "test_field":"test client 1"
}
{
  "_index" : "test_index",
  "_type" : "_doc",
  "_id" : "1",
  "_version" : 4,
  "result" : "updated",
  "_shards" : {
    "total" : 2,
    "successful" : 1,
    "failed" : 0
  },
  "_seq_no" : 3,
  "_primary_term" : 1
}

另外一个客户端，尝试基于 if_seq_no 和 if_primary_term 的数据进行修改，会更新失败：

PUT /test_index/_doc/1?if_seq_no=2&if_primary_term=1
{
  "test_field":"test client 2"
}
{
  "error": {
    "root_cause": [
      {
        "type": "version_conflict_engine_exception",
        "reason": "[1]: version conflict, required seqNo [2], primary term [1]. current document has seqNo [3] and primary term [1]",
        "index_uuid": "0jAS2GP1TPG5J8PlqGdYIQ",
        "shard": "4",
        "index": "test_index"
      }
    ],
    "type": "version_conflict_engine_exception",
    "reason": "[1]: version conflict, required seqNo [2], primary term [1]. current document has seqNo [3] and primary term [1]",
    "index_uuid": "0jAS2GP1TPG5J8PlqGdYIQ",
    "shard": "4",
    "index": "test_index"
  },
  "status": 409
}
