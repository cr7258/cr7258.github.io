        # 集群

        # 查看集群健康状态

        • cluster_name：集群名。
        • status：集群状态，共有 3 种状态，分别是 red、yellow 和 green。
          • green：是最健康的状态，说明所有的主分片和副本分片都可用。
          • yellow：主分片可用，但至少还有一个副本是未分配的。不会有数据丢失，所以搜索结果依然是完整的。可以把 yellow 想象成一个需要关注的 warnning，该情况不影响索引读写，一般会自动恢复。
          • red：至少一个主分片缺失，此时执行查询虽然部分数据仍然可以查到，但实际上已经影响到索引读写，需要重点关注。
        • timed_out：本次查询是否超时。
        • number_of_nodes：节点总数。
        • number_of_data_nodes：数据节点总数。
        • active_primary_shards：主分片的数量。
        • active_shards：主分片和副本分片的数量。
        • relocating_shards：正在迁移分片的数量。
        • initializing_shards：刚刚创建的分片数量。比如，当你刚创建第一个索引，分片都会短暂的处于 initializing 状态。这通常会是一个临时事件，分片不应该长期停留在 initializing 状态。你还可能在节点刚重启的时候看到 initializing 分片：当分片从磁盘上加载后，它们会从 initializing 状态开始。
        • unassigned_shards：未分配的分片数量。
        • delayed_unassigned_shards：由于超时设置而延迟分配的分片数。通过 index.unassigned.node_left.delayed_timeout 参数设置可以延迟分配副本的时间（此设置不会影响将副本提升为主分片），目的是当节点因为某种原因（网络断开）后很快重新加入集群，不会因为大量恢复分片而给集群带来额外的负载。
        • number_of_pending_tasks：当前集群的任务堆积情况，如果number_of_pending_tasks 数量较大，则表明 Master 在处理 task 时有点力不从心，承载的压力较大了。
        • number_of_in_flight_fetch：未完成的 fetch 操作数量。
        • task_max_waiting_in_queue_millis： 任务在队列中等待的最长时间。
        • active_shards_percent_as_number：集群种活跃分片的比例。

        Elasticsearch watermark 分为 3 个等级： cluster.routing.allocation.disk.watermark.low： 低水位，默认为 85%。Elasticsearch 不会为该节点分配新的分片。该设置对新创建索引的主分片没有影响，但是会阻止分配它们的副本。 cluster.routing.allocation.disk.watermark.high：高水位，默认为 90%。Elasticsearch 尝试将该节点上的分片迁移到其他节点上。该设置影响所有分片的分配，无论之前是否分配。 cluster.routing.allocation.disk.watermark.flood_stage：泛洪水位，默认为 95%。当某个节点的磁盘使用达到这个水平以后，这个n节点上的分片对应的 index 都会被设置为index.blocks.read_only_allow_delete，也就是只允许读操作和删除操作，这是 Elasticsearch 为了应对集群崩溃不得不采取的一个操作。

        修改 watermark，可以修改成百分比和具体数值：

        PUT _cluster/settings
{
  "transient": {
    "cluster.routing.allocation.disk.watermark.low": "100gb",
    "cluster.routing.allocation.disk.watermark.high": "50gb",
    "cluster.routing.allocation.disk.watermark.flood_stage": "10gb",
    "cluster.info.update.interval": "1m"
  }
}

        • NODE_LEFT：由于托管的节点离开集群而未分配。
        • REALLOCATED_REPLICA：确定了更好的副本位置，并导致现有副本分配被取消。
        • REINITIALIZED：当分片从开始移动回初始化，导致未分配。

        # 修改分片分配并发数

        参考资料：Cluster-level shard allocation and routing setting (opens new window)

        PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.node_concurrent_incoming_recoveries": 8,
    "cluster.routing.allocation.node_concurrent_outgoing_recoveries": 8,
    "cluster.routing.allocation.node_concurrent_recoveries": 8
  }
}

# 或者
PUT _cluster/settings
{
  "persistent": {
    "cluster.routing.allocation.node_concurrent_recoveries": 8
  }
}
        • cluster.routing.allocation.node_concurrent_incoming_recoveries：允许在节点上并发恢复传入分片的数量，传入恢复是在节点上分配目标分片（很可能是副本，除非分片正在重新定位），默认值是 2。
        • cluster.routing.allocation.node_concurrent_outgoing_recoveries：允许在节点上并发恢复传出分片的数量，传出恢复是在节点上分配源分片（很可能是主分片，除非分片正在重新定位）的恢复，默认值是 2。
        • cluster.routing.allocation.node_concurrent_recoveries 表示同时设置 cluster.routing.allocation.node_concurrent_incoming_recoveries 和 cluster.routing.allocation.node_concurrent_outgoing_recoveries。
